Manage users and access to App Searchedit

To quickly get started managing users and access to App Search, jump to Manage custom App Search mappings.

You can manage users and their access to App Search entirely within App Search. Or you can manage access to App Search separately from where you manage your users. This document therefore separates the concerns of access and users.

Users' access to App Search is determined by App Search roles and engine access lists. But you don’t directly assign App Search roles and engines to users. This is because users are managed in potentially many places, including services external to your Elastic deployment.

Instead, you manage mappings. You can rely on App Search’s default mappings, which map Elastic Stack roles to App Search roles and engine lists. In that case, you manage access to App Search indirectly, by managing Elastic Stack roles. Or you can create custom App Search mappings. In that case, you manage access to App Search directly, by mapping users (via their attributes) to App Search roles and engines.

While access is managed exclusively through App Search or the Elastic Stack, you can manage users in many places, including services external to Elastic. However, you can manage Elastic users entirely within App Search, if you’d like. This includes creating, inviting, and listing users.

Manage users' access to App Searchedit

What users can do within App Search is determined by their App Search roles and engine access lists. Users are mapped to roles and engines using role mappings. You can rely on default mappings, and therefore manage Elastic Stack roles. Or you can create custom mappings, and therefore map users to App Search roles and engines explicitly.

App Search roles and engine access listsedit

What users can do within App Search is determined by their App Search roles and engine access lists. This is a reference section for those concepts.

To access App Search, a user must have an App Search role. The specific role determines the level of access.

You can limit some users to specific engines. Use engine access lists. List of engine names. The engines the user can access. The value all within an engine access list means the user can access all current and future engines. Some roles have access to all engines regardless. So for those users engine access lists do not apply.

App Search roles are different from Elastic Stack roles. Managed separately.

App Search roles and engine access lists are specific to App Search. External IdPs have their own concepts of roles and groups. These are different. Managed separately.

App Search mappings and Elastic Stack rolesedit

You map users to App Search roles and engines based on their user attributes. The user attributes are managed wherever you manage your users and can includes fields like username, email, role. Potentially many other fields specific to the identity provider. Any fields such as role and group are separate from the App Search concepts of role and engines. You must map users to App Search roles and engines—​you can’t directly assign these values. When managing entirely within App Search, you can create users and map them to roles and engines all at the same time.

Each mapping is composed of:

User attributes.

An App Search role.

A list of App Search engines. Or the special case "all", which includes all present and future engines.

At any given time, each user matches zero or more mappings. The user is granted the role and engines of the mapping that matches.

When a user matches multiple mappings, the "most permissive" mapping applies.

Mappings are resolved …​ (when)

You can rely on default mappings, which map Elastic Stack roles to App Search roles and engines.

Or you can create custom mappings, which map users to App Search roles and engines using any user attributes you want.

Default App Search mappingsedit

These map users, based on their Elastic Stack roles, to App Search roles and engine access.

All users with the given Elastic Stack role match the mapping and receive the App Search role and engines specified in the mapping.

Enterprise Search has a feature called role-based access control (RBAC), which can be disabled or enabled at any given time. The setting applies to the entire solution, therefore App Search and Workplace Search. See Enterprise Search role-based access control (RBAC) in the Enterprise Search documentation.

Some App Search mappings are always active, while others are active only when Enterprise Search RBAC is enabled or disabled. When disabled, the default mappings are permissive. When enabled, the default mappings are restrictive.

Manage Elastic Stack rolesedit

If you are using default mappings to manage access to Enterprise Search, you don’t manage App Search mappings directly.

Instead, you manage Elastic Stack roles (since Elastic Stack roles are mapped to App Search mappings automatically).

So this section points users to the documentation that explains how to do that.

Manage these outside of App Search.

Unlike App Search roles, Elastic Stack roles can be directly assigned from the same interface where you manage the users. Or you can map Elastic Stack roles to users, based on their attributes. Just like with App Search mappings. But these are Elastic Stack mappings. Elastic Stack mappings are different from App Search mappings.

Custom App Search mappingsedit

Custom mappings allow you to explicitly map users via their user attributes to App Search roles and engines.

To manage custom mappings for your deployment, see following section.

Default mappings are limited to Elastic Stack role, but with custom mappings, you can map users based on any attribute you want.

Manage custom App Search mappingsedit

Create, modify, and delete App Search mappings. Done exclusively within App Search.

Must enable Enterprise Search RBAC. See Enterprise Search role-based access control (RBAC) in the Enterprise Search documentation.

Manage users for your Elastic deploymentedit

While access is managed exclusively through App Search or the Elastic Stack, you can manage users in many places, including services external to Elastic. These are Elastic users, not necessarily limited or unique to App Search. So we cover all the details elsewhere. See Manage users for your Elastic deployment in the Enterprise Search documentation.

However, you can manage Elastic users entirely within App Search, if you’d like. This includes creating, inviting, and listing users. See Manage users within App Search.
